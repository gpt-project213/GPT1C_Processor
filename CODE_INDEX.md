# GPT1C_Processor / AI 1C PRO — INDEX

## 1. Назначение проекта

Production-проект для автоматической обработки Excel-отчётов 1С, генерации HTML / JSON / AI-аналитики и доставки результатов через Telegram-бот.

Основной pipeline:

IMAP / локальные Excel
→ queue
→ clean xlsx
→ parser
→ report / analytics / AI
→ Telegram
→ archive / cleanup

---

## 3. Главные точки входа

### `bot/send_reports.py`
Главный Telegram-бот и оркестратор.
Функции:
- запуск polling
- меню и callback-routing
- разграничение ролей
- выдача отчётов
- запуск job/scheduler
- интеграция с AI и аналитикой
- индексация доступных HTML/JSON/PDF артефактов

### `run_pipeline.py`
Последовательный запуск обработки очереди.

### `run_pipeline_all_mp.py`
Multiprocess / orchestration вариант pipeline для пакетной обработки.

### `run_new_reports_now.py`
Ручной форсированный запуск генерации отчётов.

### `imap_fetcher.py`
Получение Excel-вложений из почты, фильтрация, сохранение в очередь.

## 4. Парсеры Excel → JSON

### `analyze_debt_excel.py`
Парсер дебиторской задолженности.
Выход:
- debt / debt_ext JSON
- клиенты
- движения по датам
- days_silence
- агрегаты

Важно:
- ключ долга: `debt`, не `closing`

### `sales_parser.py`
Парсер продаж.
Назначение:
- чтение Excel продаж
- нормализация структуры
- формирование JSON по клиентам / товарам
- извлечение менеджера

### `gross_parser.py`
Парсер валовой прибыли.
Назначение:
- чтение отчёта по валовой
- revenue / cost / gross_profit / margin
- детализация по товарам

### `expenses_parser.py`
Парсер расходов.
Назначение:
- чтение Excel расходов
- суммарные и категорийные данные

### `inventory_cost_parser.py`
Парсер себестоимости / складских остатков по партиям.
Важно:
- ориентирован на отчёт вида «Ведомость по партиям товаров…»

### `inventory.py`
Дополнительная логика по остаткам / сводным инвентарным данным.

## 11. Ключевые инварианты проекта

- Не ломать существующую production-архитектуру
- Не переписывать проект с нуля
- Не менять рабочие роли и ACL без явной команды
- Не менять действующие очереди, janitor и dedup-логику без причины
- Для debt использовать поле `debt`, не `closing`
- Для sales summary учитывать уже внедрённую накопительную логику по менеджерам
- Для всех доработок учитывать зависимости между `.env`, config, путями, ролями, логами, очередями и архивом
